# Functional Specification — CYD Home Assistant Control Display

**Target executor:** Claude Code, running on `john-ai` (Ubuntu, Ryzen 9 5950X / RTX 3090).
**Deliverable:** A working ESPHome firmware flashed over USB to a Cheap Yellow Display, acting as a wall control panel for Home Assistant.
**Author of spec:** David. **Status:** ready to build.

> **Historical note:** This is the *original* functional specification as written before the build. The shipped firmware diverges from it in several resolved ways — e.g. display model **ILI9342** (not ILI9341), **`ledc`** output (not `esp32_dac`) for the speaker, buttons use **`switch.toggle`** (not a momentary pulse), the power sensor reports **kW** (×1000 applied), and thresholds/colors have been tuned (over-power alarm now **3000 W**, label green `<200 W` / red `≥200 W`). For the current, authoritative state see **`CLAUDE.md`** ("Key Technical Decisions") and **`changelog.md`**.

---

## 1. Goal

Repurpose a CYD board (currently running NerdMiner firmware) into an ESPHome-based
Home Assistant control display that:

1. Shows the current average power consumption (`sensor.your_power`) prominently.
2. Provides three momentary buttons that trigger Home Assistant switches.
3. Beeps the onboard speaker when power stays above 1000 W for 5 s, repeating until it drops back below.

The device connects to Home Assistant via the ESPHome native API (the HA ESPHome
integration is already set up on David's instance).

## 2. Hardware

| Item | Detail |
|---|---|
| Board | `ESP32-2432S028E` — "Cheap Yellow Display" (CYD) |
| MCU | ESP32 (dual-core LX6, 240 MHz, Xtensa), 32 Mbit flash, no PSRAM |
| Display | 2.8" 320×240, **ILI9341**, SPI |
| Touch | **XPT2046** resistive, separate SPI bus |
| Audio | Onboard amplifier on **GPIO26** (ESP32 DAC1), analog |
| Backlight | GPIO21 |
| ESPHome board key | `esp32dev`, framework `arduino` |

### 2.1 Pin map (standard ESP32-2432S028)

**Display SPI bus (`spi_display`)**
| Signal | GPIO |
|---|---|
| CLK | 14 |
| MOSI | 13 |
| MISO | 12 |
| CS | 15 |
| DC | 2 |
| Backlight | 21 |

**Touch SPI bus (`spi_touch`)** — *separate bus from the display*
| Signal | GPIO |
|---|---|
| CLK | 25 |
| MOSI | 32 |
| MISO | 39 |
| CS | 33 |
| IRQ | 36 |

**Audio**
| Signal | GPIO |
|---|---|
| Speaker amp (DAC1) | 26 |

> ⚠️ The board is the **"E"** sub-variant. The pin map above is the standard CYD
> layout and is expected to match, but the audio amp population is the one
> hardware unknown. See §8 (Validation) for the speaker check.

## 3. Connectivity & deployment

- **First flash MUST be over USB.** The board currently runs NerdMiner firmware,
  which has no ESPHome OTA endpoint — so OTA to its current IP (<CYD_IP>)
  will NOT work for the initial flash. Use the USB serial port.
- The CYD is plugged into `john-ai` via USB. Identify the serial device
  (typically `/dev/ttyUSB0` or `/dev/ttyACM0`).
- After the first successful flash, OTA becomes available for subsequent updates.
- Wi-Fi 2.4 GHz only (ESP32 has no 5 GHz radio).

### 3.1 Environment setup on john-ai

1. Use a Python virtual environment; do not install ESPHome system-wide.
   ```bash
   python3 -m venv ~/esphome-cyd-venv
   source ~/esphome-cyd-venv/bin/activate
   pip install --upgrade esphome
   ```
2. Ensure the user running ESPHome can access the serial port (add to the
   `dialout` group, or run with appropriate permissions):
   ```bash
   sudo usermod -aG dialout "$USER"   # log out/in afterwards, or use newgrp
   ```
3. Confirm the port:
   ```bash
   ls -l /dev/ttyUSB* /dev/ttyACM* 2>/dev/null
   dmesg | tail -n 20   # watch which device appears on plug-in
   ```

## 4. Configuration & secrets

Create a `secrets.yaml` in the project directory with:

```yaml
wifi_ssid: "<SSID>"
wifi_password: "<PASSWORD>"
api_encryption_key: "<32-byte base64 key>"
```

- The API encryption key can be generated by ESPHome (the dashboard/wizard offers
  one) or with:
  ```bash
  python3 -c "import secrets,base64; print(base64.b64encode(secrets.token_bytes(32)).decode())"
  ```
- **Claude Code must NOT invent or commit real Wi-Fi credentials.** Leave clear
  placeholders and instruct David to fill them in, OR prompt David for them at
  build time. Do not hard-code secrets into the main YAML.

## 5. Functional requirements

### FR-1 — Power display
- Import `sensor.your_power` from Home Assistant via the `homeassistant`
  sensor platform.
- Display the value large and centered, formatted as whole watts, e.g. `742 W`.
- Update live on every received value.
- Assume the source reports in plain watts. (If it reports kW, conversion must be
  added — flag to David, do not silently assume.)

### FR-2 — Control buttons (three, momentary pulse)
Each button, on tap, performs an **immediate momentary pulse**: call
`switch.turn_on`, wait **500 ms**, then `switch.turn_off`. No confirmation step.

| Button label | Entity |
|---|---|
| Garage OPENEN | `switch.your_garage_open` |
| Garage SLUITEN | `switch.your_garage_close` |
| Deurslot WACHTKAMER | `switch.your_door_lock` |

> These are assumed to be `switch.*` domain entities. If any turns out to be a
> `cover.*` or `lock.*` entity, the service call must change accordingly — flag
> to David rather than guessing.

### FR-3 — Over-power beep
- Threshold: **1000 W**.
- The alarm **arms only after power has been continuously above 1000 W for 5 s**
  (debounce — brief spikes must not trigger it).
- While armed, the speaker emits a **short beep repeating every ~3 seconds**.
- The alarm **stops immediately** when power drops below 1000 W.
- Beep is produced via the GPIO26 DAC using the ESPHome `rtttl` component.

### FR-4 — Visual alarm cue (nice-to-have, include it)
- While the alarm is active, render the power number in red (`0xFF3B30`);
  otherwise white.

### FR-5 — Backlight
- Backlight on full brightness at boot, always on.

## 6. Non-functional requirements
- All logic lives on-device in ESPHome. No Home Assistant automations are created.
- Metric units only.
- Dutch button labels as specified above.
- Code must be readable and commented; use `substitutions:` for the entity IDs,
  threshold, and pulse length so they are easy to change.
- The firmware name/hostname: `cyd-control`.

## 7. Reference implementation (starting point)

A known-good ESPHome YAML is provided alongside this spec as
`cyd-ha-control.yaml`. Claude Code should use it as the basis. It already
implements FR-1 through FR-5. The agent's job is to:

1. Place it in the project directory.
2. Create `secrets.yaml` (with David's input for credentials).
3. Compile it.
4. Flash over USB.
5. Validate against §8 and fix any board-specific issues.

If the reference YAML is unavailable, build from the requirements above; the key
structural points are: two separate `spi:` buses, `ili9xxx` (model `ILI9341`)
display, `xpt2046` touchscreen, `homeassistant` sensor import, `lvgl` UI,
`esp32_dac` output on GPIO26 feeding `rtttl`, a template `binary_sensor` with a
`delayed_on: 5s` filter, and a `while` loop script for the repeating beep.

## 8. Build, flash & validation procedure

### 8.1 Compile
```bash
source ~/esphome-cyd-venv/bin/activate
cd <project-dir>
esphome compile cyd-ha-control.yaml
```
Resolve any compile errors before flashing.

### 8.2 Flash over USB
```bash
esphome run cyd-ha-control.yaml --device /dev/ttyUSB0
```
(Substitute the correct port. The board may need to be put into boot mode —
on most CYDs this is automatic via the USB-UART chip; if not, hold BOOT while
plugging in / pressing reset.)

### 8.3 Validation checklist
Work through these and fix issues:

1. **Boot & Wi-Fi:** Device boots, joins Wi-Fi, appears in the ESPHome logs.
   Watch live logs: `esphome logs cyd-ha-control.yaml`.
2. **HA API:** Device connects to Home Assistant (ESPHome integration shows it
   online). `sensor.your_power` value appears on screen.
3. **Display orientation/colour:** Text is upright and legible.
   - If colours look inverted (reds appear cyan), set `invert_colors: true` on
     the display and reflash.
   - If orientation is wrong, adjust `rotation`.
4. **Touch accuracy:** Tap each button; confirm the tap registers on the button
   you touched.
   - If taps are offset, recalibrate the XPT2046: temporarily log raw touch
     coordinates, touch the four corners, and update the `calibration:` block
     (`x_min/x_max/y_min/y_max`) and `transform:` (`swap_xy/mirror_x/mirror_y`).
5. **Button actions:** Each button pulses its switch on then off after 500 ms.
   Verify in HA logbook that the switch toggled on→off for each of the three
   entities.
6. **Speaker check (the hardware unknown):** Trigger a beep to confirm GPIO26
   audio works. Easiest temporary test — add a button in HA or a boot-time
   `rtttl.play` to sound a note. If silent:
   - Confirm GPIO26 is the amp pin on this "E" unit.
   - Confirm `esp32_dac` output + `rtttl` gain are set; raise `gain:`.
   - If still silent, report to David that the amp may be unpopulated.
7. **Beep logic:** Force `sensor.your_power` above 1000 W (in HA, set the
   sensor or its source) and confirm: nothing happens for the first 5 s, then a
   beep repeats every ~3 s; drop below 1000 W and confirm it stops immediately.

### 8.4 Done criteria
All of §8.3 pass. Report final port used, any YAML changes made (especially
`invert_colors`, `rotation`, touch calibration, and speaker gain), and confirm
OTA is now available for future updates.

## 9. Things to flag to David rather than guess
- Any entity that is not actually a `switch.*` domain.
- `sensor.your_power` reporting in kW instead of W.
- Speaker silent after the GPIO26/gain checks (possible unpopulated amp).
- Touch calibration values that differ substantially from the defaults.
- Wi-Fi credentials — prompt for these; never commit real ones.
